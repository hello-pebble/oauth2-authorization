# 🏔️ Challenge 03: 비정상 트래픽의 파동과 엔진 보호

## 1. 🚨 The Trigger (발단: 상황)
한정판 굿즈 판매 이벤트가 시작되자마자, 선량한 유저들 사이로 수만 개의 **악의적인 매크로 봇(Bot)**이 들이닥쳤습니다. 이들은 인간이 할 수 없는 속도로 수만 번의 로그인을 시도하며 서버의 모든 입구를 점령해버렸고, 정작 정직하게 기다리던 일반 유저들은 로그인 버튼 한 번 눌러보지 못하고 **'서비스 불가'** 메시지만 마주하는 극도로 불공정한 상황이 벌어졌습니다.

## 2. 💣 The Pain Point (문제 정의)
**"엔진이 아무리 빨라도, 쓰레기 데이터가 입구를 막으면 무용지물이다."**
인증 엔진의 성능을 최적화하는 것보다 더 중요한 것은, 엔진이 '가치 있는 요청'에만 집중할 수 있도록 비정상 트래픽을 입구에서 걸러내는 **방호벽**의 부재였습니다.

## 3. 🛡️ The Constraints (제약 사항)
- **Zero Latency for Normal Users**: 정상 유저의 요청은 지연 없이 통과시켜야 함.
- **Fairness (공정성)**: 먼저 온 유저가 반드시 먼저 처리되어야 함 (FIFO).
- **Resource Protection**: 비정상 요청이 CPU나 메모리 자원을 점유하지 못하게 즉각 차단해야 함.

## 4. ⚔️ The Weapon (기술 선택: Rate Limiting & Virtual Waiting Room)
우리는 다중 방어 체계를 구축했습니다.

| 선택 기술 | 해결하려는 핵심 이유 |
| :--- | :--- |
| **Bucket4j (Rate Limit)** | IP별 요청 속도를 제한하여 매크로 및 DDoS 공격을 입구(Filter)에서 즉시 차단 |
| **Redis ZSET (Waiting Room)** | 서버 수용량 초과 시 요청을 버리지 않고 '가상 대기열'에 담아 순차적으로 처리 |
| **Batch Processing** | 인증(BCrypt) 연산의 TPS를 강제로 조절하여 서버 가용성(Resilience) 유지 |
| **BCrypt Migration** | Argon2 대비 CPU 집약적인 특성을 활용하여, 리소스 예측 가능성 확보 |

## 5. 📉 The Proof (검증)
- **방어율**: 비정상 매크로 요청의 **98% 이상을 Filter 레이어에서 즉각 차단** (429 Too Many Requests).
- **공정성**: 10만 명의 동시 접속 상황에서도 유저들이 **서버 진입 순번을 실시간으로 확인**하며 예측 가능한 대기 경험 획득.
- **안정성**: 트래픽 스파이크 시에도 서버의 CPU 사용량이 **80% 선을 넘지 않도록 제어**되어 시스템 전체 붕괴 예방.
