# ⚡ High-Traffic Login Strategy: Why & How

본 문서는 티켓팅이나 선착순 이벤트와 같은 **폭발적 트래픽(Traffic Spike)** 상황에서도 사용자 로그인 단계에서 지연이 발생하지 않도록 하기 위한 전략적 선택을 기록합니다.

---

## 1. 🎯 해결하려는 핵심 문제 (The Problem)
*   **로그인 병목**: 이벤트 시작 직후 수만 명의 사용자가 동시에 접속할 때, 기존 세션 방식은 서버 메모리 부족이나 세션 저장소(Redis 등)의 I/O 병목으로 인해 로그인 화면부터 '무한 로딩'이 발생함.
*   **목표**: 로그인 요청이 쏟아져도 서버가 '상태(State)'를 들고 있지 않음으로써, 서버를 늘리는 만큼 정비례하게 접속자를 처리할 수 있는 구조를 만드는 것.

---

## 2. 🏗️ 왜 'Method A(Stateless JWT + Kotlin)'인가?

### A. Stateless 아키텍처 (No More Session)
*   **선택 이유**: 세션 방식은 로그인을 확인하기 위해 매번 서버가 저장소를 찔러봐야 하지만, **JWT**는 서버가 토큰의 유효성을 스스로 검증함. 
*   **결과**: 데이터베이스나 외부 저장소 조회를 최소화하여 응답 속도를 극대화하고, 서버 대수를 10대, 100대로 늘려도 공유 세션에 대한 부담 없이 즉각 대응 가능함.

### B. Kotlin & Coroutine (Non-blocking Ready)
*   **선택 이유**: 대용량 트래픽 상황에서 'Thread'는 자원 낭비임. 사용자가 대기하는 동안 서버 자원을 점유하지 않아야 함.
*   **결과**: 향후 **Kotlin Coroutines**를 도입하여, 수천 개의 동시 연결을 적은 메모리로 처리할 수 있는 기반을 마련함. (Java 대비 획기적인 동시성 처리 효율)

---

## 3. 🔍 현재 상태 및 기술적 근거 (Current Status)

| 병목 지점 | 현재 해결책 | 기대 효과 |
| :--- | :--- | :--- |
| **세션 동기화** | **Stateless JWT** | Redis I/O 부하 제거, 서버 수평 확장성 확보 |
| **토큰 탈취 대응** | **RTR (Refresh Token Rotation)** | 보안을 유지하면서도 성능을 포기하지 않음 |
| 코드 복잡도 | **Kotlin Idioms** | 성능 관련 핵심 로직(보안 필터)의 유지보수성 향상 |

---

## ⚖️ Trade-offs: 무엇을 얻고 무엇을 포기했는가?

대용량 트래픽에서의 '로그인 무한 로딩'을 해결하기 위해 우리는 의도적으로 다음과 같은 균형점을 선택했습니다.

### 1. 확장성(Scalability) vs 즉각적 제어(Real-time Control)
*   **얻은 것**: 서버가 세션을 들고 있지 않으므로, 트래픽이 몰릴 때 서버를 수평적으로 무한히 늘려도 성능 저하가 없음.
*   **포기한 것**: 한번 발행된 Access Token은 서버가 중앙에서 즉시 무효화하기 어려움. 즉, '강제 로그아웃' 처리에 약간의 시차가 발생할 수 있음.
*   **전략**: Access Token의 만료 시간을 짧게 설정하여 보안 리스크를 최소화함.

### 2. 보안(Security) vs 성능(Performance)의 타협점 (RTR)
*   **얻은 것**: Refresh Token Rotation(RTR)을 통해 토큰 탈취 시나리오에 강력하게 대응함.
*   **포기한 것**: 완전한 Stateless를 지향하지만, RTR 검증을 위해 Refresh Token은 Redis 저장소를 조회해야 함.
*   **전략**: 매 요청마다 확인하는 Access Token은 저장소 조회 없이 서버 자체 검증으로 처리하여, 전체적인 응답 속도(Latency)는 세션 방식보다 훨씬 빠르게 유지함.

### 3. 생산성(Modernity) vs 설정 복잡도(Compatibility)
*   **얻은 것**: Kotlin & Coroutines 기반의 비동기 처리 준비로 적은 자원으로도 많은 요청을 처리함.
*   **포기한 것**: Kotlin의 기본 제약(final 클래스 등)으로 인해 Spring Framework와의 통합 설정(all-open 등)이 Java보다 까다로움.
*   **전략**: 초기 설정의 복잡함보다는 런타임의 안정성과 대용량 처리 효율에 더 높은 가치를 둠.

---

## 5. 결론: "로그인 로딩 0초"를 위한 방향
우리는 단순히 '로그인 기능'을 만드는 것이 아니라, **"10만 명이 동시에 접속해도 서버가 죽지 않고 즉시 토큰을 발행해주는 인증 엔진"**을 지향함. 지금의 Kotlin 전환과 Stateless 구조는 그 폭발적인 트래픽을 견디기 위한 최소한의 설계이자 필수적인 기반임.
