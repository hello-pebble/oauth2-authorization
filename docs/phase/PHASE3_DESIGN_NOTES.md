# Phase 3 설계 노트: OAuth2.0 통합의 실무적 결정과 한계

**작성자**: 백엔드 개발자 (4년 차)
**일자**: 2026-03-31

## 1. 개요
본 문서는 Phase 3 OAuth2.0 통합 과정에서 내린 기술적 결정의 배경과 현재 구조가 가진 한계점을 솔직하게 기록합니다. 이는 단순한 기능 구현을 넘어, 유지보수와 확장성을 고려한 "현실적인 타협점"을 공유하기 위함입니다.

## 2. 주요 설계 결정 및 사유 (Rationale)

### 2.1. 독립적 계정 체계 (Provider-based Identity)
- **결정**: 동일한 이메일이라도 Provider(Google, GitHub)가 다르면 별개의 유저로 생성함.
- **이유**: 계정 통합(Account Linking) 로직은 이메일 인증 절차 등 복잡한 비즈니스 프로세스가 수반됨. MVP 단계에서는 데이터 무결성을 위해 `provider` + `providerId`를 유니크 키로 가져가는 것이 가장 안전하고 빠른 경로라고 판단함.

### 2.2. Cookie-based JWT 전략
- **결정**: Access/Refresh 토큰 모두 HttpOnly 쿠키로 전달.
- **이유**: 프론트엔드 개발자가 로컬 스토리지 보안에 신경 쓰지 않아도 XSS를 방어할 수 있는 구조를 제공하고 싶었음. 또한, 기존 세션 방식의 경험을 유지하면서도 서버는 Stateless하게 가져갈 수 있는 "익숙하면서 강력한" 방식임.

## 3. 현재 구조의 한계 및 단점 (Limitations)

### 3.1. 트랜잭션 범위 문제 (DB Connection Starvation)
- **현상**: `CustomOAuth2UserService.loadUser` 전체에 `@Transactional`이 걸려 있음.
- **리스크**: 소셜 서비스의 UserInfo API를 호출하는 동안 DB 커넥션을 점유하게 됨. 트래픽 급증 시 커넥션 풀 부족 현상이 발생할 수 있는 잠재적 위험 요소임.

### 3.2. 하드코딩된 설정값
- **현상**: 쿠키 만료 시간 등이 코드 내에 숫자로 박혀 있음.
- **리스크**: 만료 정책 변경 시 소스 코드를 수정하고 재배포해야 함. `JwtProvider`와의 싱크가 깨질 위험이 큼.

### 3.3. 비즈니스 로직의 파편화
- **현상**: 유저 생성 로직이 서비스 레이어에 절차적으로 노출되어 있음.
- **리스크**: 유저 생성 규칙이 변경될 때 여러 곳을 수정해야 하며, 도메인 모델의 자율성이 떨어짐.

## 4. 향후 개선 계획 (Roadmap)
1. **계정 통합 시스템**: 로그인 후 설정 페이지에서 다른 소셜 계정을 연동하는 기능 추가.
2. **인프라 최적화**: 외부 API 호출과 DB 작업을 철저히 분리하여 트랜잭션 효율화.
3. **보안 고도화**: 쿠키의 `SameSite` 속성 세밀화 및 CSRF 토큰 도입 검토.
